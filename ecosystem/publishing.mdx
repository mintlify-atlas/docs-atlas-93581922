---
title: "Publishing packages"
description: "Publish your Gleam package to Hex.pm"
---

Gleam packages are published to [Hex.pm](https://hex.pm), the package repository shared with the Erlang and Elixir communities. Publishing makes your package available to any Gleam, Erlang, or Elixir project.

## Requirements

Before publishing, your `gleam.toml` must include a `description` and at least one entry in `licences`. Hex will reject packages that are missing either field.

```toml gleam.toml
name = "my_package"
version = "1.0.0"
description = "A concise description of what my package does"
licences = ["Apache-2.0"]
```

Your project must also have a non-empty `README.md` that is not the default template generated by `gleam new`.

<Note>
  All dependencies listed in `gleam.toml` must be Hex packages when publishing. Git and local path dependencies are not permitted in published releases.
</Note>

## Semantic versioning

Gleam packages follow [semantic versioning](https://semver.org): `MAJOR.MINOR.PATCH`.

- **Patch** — bug fixes only, no API changes.
- **Minor** — new functionality added in a backwards-compatible way.
- **Major** — breaking changes to the public API.

<Warning>
  Semantic versioning does not apply to `0.x.x` releases. Publishing a `0.x.x` version will prompt a confirmation, because consumers cannot rely on the stability guarantees that semver provides. Publish `1.0.0` when your package is ready for production use.
</Warning>

## Authentication

`gleam publish` requires a Hex account. Authenticate interactively:

```bash
gleam hex authenticate
```

This stores credentials locally. Alternatively, set the `HEXPM_API_KEY` environment variable with a Hex API key. This is the recommended approach for CI/CD pipelines:

```bash
HEXPM_API_KEY=your_api_key gleam publish
```

## Publishing

<Steps>
  <Step title="Check your package">
    Run `gleam build` and `gleam test` to confirm the package compiles cleanly and all tests pass. The compiler also blocks publishing if any module contains a `todo` expression or an `echo` debug call.
  </Step>
  <Step title="Publish">
    ```bash
    gleam publish
    ```

    The compiler will:
    1. Build the project in production mode.
    2. Show you the list of files that will be included in the release.
    3. Ask for confirmation before uploading.
    4. Upload the package tarball and generated HTML documentation to Hex.
  </Step>
  <Step title="Tag the release">
    After publishing, create a Git tag so that source code links in the generated documentation work correctly:

    ```bash
    git tag v1.0.0
    git push origin v1.0.0
    ```
  </Step>
</Steps>

### Skipping confirmation

Pass `-y` (or `--yes`) to publish without the interactive confirmation prompt. Useful in automated workflows:

```bash
gleam publish -y
```

### Replacing the latest release

If you need to overwrite the most recently published version (for example, to fix a packaging error immediately after release), use `--replace`:

```bash
gleam publish --replace
```

<Warning>
  `--replace` only works within a short window after the original release. It cannot be used to replace older versions.
</Warning>

## Documentation

Documentation is built from doc comments in your Gleam source and published to [HexDocs](https://hexdocs.pm) automatically when you run `gleam publish`.

Build docs locally to preview them before publishing:

```bash
# Build HTML docs in build/dev/docs
gleam docs build

# Open the docs in a browser after building
gleam docs build --open
```

Publish docs independently (without a new package release):

```bash
gleam docs publish
```

## Retiring a release

Retiring marks a release as unsuitable for new use without removing it. Existing users are warned; new projects are steered away from it.

```bash
gleam hex retire --package my_package --version 1.2.3 --reason security
```

Valid reasons: `other`, `invalid`, `security`, `deprecated`, `renamed`.

To reverse a retirement:

```bash
gleam hex unretire --package my_package --version 1.2.3
```

## Reverting a release

Reverting removes a release from Hex entirely. This is a more drastic action than retiring and should only be used when a release must be deleted (for example, when sensitive data was accidentally included).

```bash
gleam hex revert --package my_package --version 1.2.3
```

## Package ownership

### Add an owner

```bash
gleam hex owner add my_package --user colleague@example.com
```

The `--level` flag controls the ownership level (`maintainer` by default, or `full`).

### Transfer ownership

```bash
gleam hex owner transfer --package my_package --to new_owner
```

<Warning>
  Transferring ownership is irreversible. The new owner gains full control of the package on Hex.
</Warning>
